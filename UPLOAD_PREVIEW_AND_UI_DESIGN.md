# 上传实时预览与上传页 UI 设计方案

本文档用于指导“上传过程中展示实时预览（PDF 首页缩略图）”与上传页整体验证升级的代码落地。  
目标不是增加一个孤立功能点，而是把上传页从“可用”推进到“完成度高、视觉统一、具备产品感”的状态。

适用范围：

- `apps/web`
- 上传领域 `upload domain`
- `useInstance() -> instance.upload.xxx(...)`
- 样式源文件：[apps/web/src/global.css](/Users/realrong/Desktop/AI_RESUME/apps/web/src/global.css)

必须继续遵循的约束：

- 组件必须严格基于 `shadcn/ui`
- 样式必须保持黑白极简
- 颜色只能使用语义 token
- 页面不允许直接散写业务请求
- 页面不允许直接维护分散业务状态
- 缩略图、进度、基础提取信息都必须进入 `upload domain`

---

## 1. 目标

本轮设计要同时解决四件事：

1. 满足 `REQUIREMENT.md` 中“上传过程中展示实时预览（PDF 首页缩略图）”的加分项。
2. 统一上传页的视觉语言，避免边框嵌套边框、调试面板感和布局碎片化。
3. 提升上传过程的感知质量，让用户在文件刚进入队列时就获得明确反馈。
4. 为后续候选人详情页 PDF 能力和上传页交互增强打基础。

---

## 2. 技术方案结论

### 2.1 选型结论

对于“上传过程中实时展示 PDF 首页缩略图”，**不建议为了这一点引入 `react-pdf`**。  
当前目标只是：

- 本地选择文件后立即生成缩略图
- 只渲染第一页
- 只用于上传队列中的轻量视觉预览

因此更合理的方案是：

- 使用 `pdfjs-dist`
- 读取用户本地 `File`
- 渲染 PDF 第 1 页到离屏 `canvas`
- 导出为缩略图 URL
- 将缩略图状态写入 `upload domain`

### 2.2 推荐依赖

本轮推荐的依赖分层如下。

必需：

- `pdfjs-dist`

强烈建议：

- `react-dropzone`

可选但推荐：

- `framer-motion`
- `sonner`

理由如下：

- `pdfjs-dist`：负责本地 PDF 首页解析与缩略图渲染
- `react-dropzone`：负责拖拽上传、文件过滤、拖拽态反馈、批量选择体验
- `framer-motion`：用于上传卡片插入、缩略图出现、状态切换的轻量过渡
- `sonner`：用于格式错误、文件过大、上传失败、上传完成等即时反馈

如果坚持最小依赖集合，本轮至少需要：

- `pdfjs-dist`
- `react-dropzone`

## 3. 缩略图实现方案

### 3.1 总体原则

缩略图必须在前端本地生成，而不是等待上传完成后再请求后端返回。

这样做的原因：

- 更符合“上传过程中实时预览”的要求
- 用户选中文件后立即有反馈
- 不增加后端截图任务
- 不增加 Supabase 额外对象写入成本
- 不依赖后端链路成功与否

### 3.2 建议流程

建议上传流程改成以下顺序：

1. 用户拖入或选择本地 PDF 文件
2. 前端立即为每个文件创建本地上传占位项
3. 同步触发本地缩略图生成任务
4. 缩略图生成成功后更新对应上传项的预览状态
5. 并行触发后端上传请求
6. 上传返回正式 `uploadId` 后，用真实记录替换本地占位项
7. 继续通过 SSE 更新解析进度、提取结果与最终状态

### 3.3 PDF 首页缩略图生成流程

使用 `pdfjs-dist` 的建议处理流程：

1. 读取浏览器本地 `File`
2. 转成 `ArrayBuffer`
3. 用 `pdfjs-dist` 读取 PDF 文档
4. 获取第 1 页
5. 以固定目标宽度计算 viewport
6. 使用离屏 `canvas` 渲染第一页
7. `canvas.toBlob(...)`
8. 用 `URL.createObjectURL(blob)` 生成缩略图 URL
9. 回写到 `upload domain`

### 3.4 缩略图尺寸建议

为了性能和视觉统一，缩略图不要使用原始尺寸。

建议：

- 宽度：`120px` 到 `160px`
- 高度：按比例自动计算
- 只渲染一倍像素密度版本即可

这样做的优点：

- 批量上传时性能稳定
- 队列布局更整齐
- 图片足够辨认，不会过大抢占版面

### 3.5 内存与资源释放

这一点必须明确，否则批量上传时很容易出现内存泄漏。

需要释放的资源：

- 本地 PDF 的 `objectURL`
- 缩略图 Blob 的 `objectURL`
- `pdfjs-dist` 生成过程中的中间引用

需要释放的时机：

- 上传项被移除时
- 页面卸载时
- 队列清空时
- 同一上传项缩略图被重新生成替换时

### 3.6 失败兜底

缩略图生成失败不能阻断上传流程。

可能失败的场景：

- PDF 结构异常
- 文件损坏
- 加密 PDF
- `pdf.js` 解析失败

失败后的策略：

- 上传仍继续
- 上传项显示默认 PDF 占位图
- 状态字段记录为 `thumbnail_failed`
- UI 不弹硬错误，只在必要时显示轻量提示

结论：

- “缩略图失败”不等于“上传失败”

## 4. 前端领域建模建议

### 4.1 为什么必须进入 upload domain

缩略图不能作为页面局部 `useState` 临时存放。  
否则会直接破坏你当前已经建立的结构约束：

- 页面散乱状态会回流
- 真实上传项与本地缩略图状态难以对齐
- 占位项替换为正式上传项时容易丢失数据
- 上传页以外无法复用该预览信息

因此缩略图必须作为上传领域的一部分进入 `upload domain`。

### 4.2 建议新增字段

建议在 `UploadQueueItem` 中增加这类字段：

```ts
type UploadThumbnailStatus =
  | "idle"
  | "generating"
  | "ready"
  | "failed";

type UploadPreview = {
  fileObjectUrl?: string;
  thumbnailUrl?: string;
  thumbnailStatus: UploadThumbnailStatus;
  thumbnailError?: string;
  pageCount?: number;
};
```

并将其合并到上传项中：

```ts
type UploadQueueItem = {
  uploadId: string;
  fileName: string;
  progress: number;
  status: "queued" | "uploading" | "parsing" | "extracting" | "completed" | "failed";
  candidateId?: string;
  error?: string;
  preview?: UploadPreview;
  events?: Array<...>;
  partialExtraction?: {
    basic?: Record<string, unknown>;
  };
};
```

### 4.3 建议新增 actions

建议新增以下上传领域 action：

- `createLocalUploadPlaceholders`
- `setUploadThumbnailGenerating`
- `setUploadThumbnailReady`
- `setUploadThumbnailFailed`
- `replaceLocalUploadsWithServerUploads`
- `disposeUploadPreviewResources`

### 4.4 建议新增 instance 编排职责

建议让 `instance.upload` 负责以下事情：

- 创建本地上传占位项
- 生成缩略图
- 发起后端上传
- 替换正式上传项
- 订阅 SSE
- 统一释放本地资源

页面只负责：

- 触发文件选择
- 渲染上传状态
- 调用 `instance.upload.createUploads(...)`

页面不负责：

- 直接调用 `pdfjs-dist`
- 直接管理 `objectURL`
- 直接处理上传项的状态替换

## 5. 上传页 UI 重构建议

### 5.1 当前问题

当前上传页虽然已经比第一版干净，但仍有这些问题：

- 页面区块过碎
- 信息流存在重复
- “最新处理进度”和“上传列表”信息重叠
- 上传区和队列区的视觉主次不够明确
- 上传队列还不够像正式产品中的“媒体列表”
- 预览能力缺失，页面重心不足

### 5.2 新的页面结构建议

建议将上传页重构为三层结构。

#### 第一层：顶部摘要区

保留 3 到 4 个简洁指标：

- 总数
- 处理中
- 已完成
- 失败

要求：

- 不要做得太大
- 不要过度强调
- 作为页面的运行状态概览即可

#### 第二层：主上传面板

这是页面的视觉核心。

建议结构：

- 左上：标题 + 一句简短说明
- 右上：`选择文件` 按钮
- 中部：主拖拽区
- 底部：规则提示

内容建议：

- 仅支持 PDF
- 单份大小限制
- 支持批量上传
- 当前已选择几份文件

视觉建议：

- 只保留一层主面板边界
- 拖拽区使用 `bg-muted` 建立层级
- 拖拽激活时只变化背景和 `ring`
- 不再使用“外层卡片 + 内层虚线边框 + 内层信息块”的多层边框结构

#### 第三层：上传队列

上传队列建议改成“媒体列表”风格，而不是一张张小卡片。

每一条上传项建议结构如下：

- 左侧：PDF 首页缩略图
- 中部上方：文件名
- 中部下方：候选人基础信息或“识别中”
- 右侧：状态 badge + 进度百分比
- 底部：进度条 + 错误信息 / 提取摘要

要求：

- 整个列表外层只保留一层容器
- 列表项之间使用 `divide-y`
- 单条项内部使用留白和背景区分层级
- 不再让每条记录都有完整外边框

## 6. 上传页视觉优化建议

### 6.1 设计关键词

建议这一页的视觉方向是：

- 安静
- 明确
- 专业
- 有秩序
- 不炫技

### 6.2 组件层建议

继续严格使用 `shadcn` 体系：

- `Card`
- `Button`
- `Badge`
- `Progress`
- `Skeleton`
- `Separator`
- `ScrollArea`

如果需要补充：

- `Tooltip`
- `Dialog`

### 6.3 拖拽区建议

拖拽区应该具有明确的状态变化：

- 默认态：浅 `muted` 背景
- hover 态：稍强的 `muted`
- drag active：`ring-1` 或 `ring-2`
- reject 态：轻量 `destructive` 提示

不要做的事：

- 不要加太多边框
- 不要做夸张动画
- 不要把文案堆很多

### 6.4 缩略图区域建议

缩略图建议：

- 宽度固定
- 比例稳定
- 白底
- 轻边界或内阴影都不要过强

可以加的细节：

- 文件页数角标
- PDF 标识
- 上传完成后轻量完成标记

### 6.5 状态展示建议

状态要统一中文化，并映射成简洁文案：

- `上传中`
- `解析中`
- `提取中`
- `已完成`
- `失败`

不要直接显示：

- `uploading`
- `parsing`
- `extracting`
- `completed`
- `failed`

### 6.6 信息展示建议

基础提取信息建议只显示：

- 姓名
- 邮箱
- 电话
- 城市

不要默认展开原始 JSON。  
如果后续想保留更详细信息，建议使用：

- `Dialog`
- 或“展开更多”交互

## 7. 骨架屏与加载态建议

这部分是上传页美观度和完成度的重要来源，建议本轮一并落地。

### 7.1 为什么需要骨架屏

上传页里会出现多个异步阶段：

- 文件刚进入队列
- 缩略图生成中
- 上传请求进行中
- 解析进行中
- 基础信息提取中

如果这些阶段没有过渡，页面会显得突兀和不稳定。

### 7.2 建议的骨架层级

#### 缩略图骨架

在缩略图生成阶段显示固定比例骨架块：

- 比例接近 PDF 页
- 使用 `Skeleton`
- 不要闪烁过强

#### 基础信息骨架

当 `partialExtraction.basic` 尚未返回时：

- 显示 2 到 4 行短骨架
- 模拟姓名、邮箱、城市等信息区

#### 队列项骨架

当本地占位项创建成功，但后端正式 `uploadId` 尚未返回时：

- 整条上传项可显示为“可交互但未完成”的骨架状态
- 不要完全空白

### 7.3 视觉建议

骨架屏要低噪音：

- 使用 `bg-muted`
- 不要使用彩色 shimmer
- 不要过于强烈的 loading 动画

目标是：

- 看起来稳定
- 不是花哨演示效果

## 8. 动效建议

### 8.1 建议使用场景

如果引入 `framer-motion`，建议只用在以下三个地方：

1. 上传项插入列表
2. 缩略图从骨架切换到真实图片
3. 状态 badge 切换

### 8.2 不建议使用的场景

- 不要给整页做大范围浮动动画
- 不要给每个数字做无意义动效
- 不要让上传列表像聊天应用一样跳动

### 8.3 动效原则

- 快
- 轻
- 不打断任务流

## 9. 交互增强建议

### 9.1 建议补充的交互

建议本轮或下一轮补充以下体验：

- 拖拽进入时的明确高亮态
- 文件格式不合法时 toast 提示
- 文件超大小时 toast 提示
- 上传失败时列表项级错误提示 + toast
- 批量上传完成后的轻量成功反馈

### 9.2 缩略图交互建议

缩略图可以支持：

- 点击查看大图
- hover 显示文件名与页数

但不建议在上传页内做完整 PDF 阅读。

上传页职责应保持克制：

- 确认文件
- 观察状态
- 快速发现异常

## 10. 分阶段落地建议

建议按这个顺序推进，而不是一次写满。

### 阶段一：结构改造

- 合并上传页区块结构
- 调整主上传区和上传队列布局
- 去掉边框嵌套边框
- 建立单层容器 + 分隔线的视觉结构

### 阶段二：缩略图能力

- 引入 `pdfjs-dist`
- 实现本地 PDF 首页缩略图
- 写入 upload domain
- 做资源释放

### 阶段三：加载与反馈

- 接入骨架屏
- 处理缩略图生成中状态
- 处理格式错误、上传失败、解析失败反馈

### 阶段四：精修

- 接入轻量动效
- 优化缩略图信息密度
- 完善 hover 和空态体验

## 11. 最终验收标准

本轮完成后，上传页至少应达到：

- 满足 `REQUIREMENT.md` 的 PDF 首页缩略图加分项
- 用户选中文件后立即看到实时缩略图
- 缩略图失败不影响上传主流程
- 上传页没有明显边框嵌套边框问题
- 上传列表呈现为统一、克制、正式的媒体列表
- 用户可见状态全部中文化
- 骨架屏和空态完整
- 整体风格保持黑白极简，符合当前项目的设计方向

---

## 12. 推荐结论

结论明确如下：

- 上传页实时缩略图应采用 `pdfjs-dist`，不必为了缩略图引入 `react-pdf`
- 上传交互建议引入 `react-dropzone`
- 本轮值得一并补充 `Skeleton`、轻量 toast、必要的状态过渡
- 上传页应该从“上传控件 + 日志列表”升级为“上传工作台”

如果进入代码落地，优先级建议是：

1. 页面结构重构
2. `upload domain` 增加 preview 状态
3. `instance.upload` 接入缩略图生成
4. 骨架屏与错误反馈
5. 动效精修
